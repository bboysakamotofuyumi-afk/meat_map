## 運用メモ（単一ファイル管理）

- 要件はこのファイル1本で管理し、ブランチごとの差分も冒頭に追記する。
- ブランチを切ったら、ブランチ名・日付・差分メモをここに追加する。
- 現行: make-a-big-promise（MVPベース、差分メモなし）

## 実行手順（CSV生成の最小フロー）

- 前提: `HOTPEPPER_API_KEY` を環境変数または `.env` に設定。
- コマンド例: `python -m meatmap.cli --output output/meatmap.csv`
- ランクフィルタ: デフォルトは S/A のみを出力。`--include-rank-b` や `--include-rank-c` で範囲を広げられる。
- マップデモ: `python -m http.server 8000` などでルートをホストし、`http://localhost:8000/docs/map_demo.html?csv=/output/meatmap.csv` を開くと CSV を地図表示できる（クエリを省略すると `/output/meatmap.csv` を読む）。
- GitHub Pages 用コピー: `--copy-to-docs` を付けて実行すると `docs/output/meatmap.csv` にもコピーされる（Pagesでそのまま参照可能）。

## スコア閾値（ルールベース）

- S: 80 以上
- A: 60 以上
- B: 45 以上
- C: 上記未満
- CSV出力のデフォルト対象は S/A。閾値は `meatmap/scoring.py` で定数管理。

### 1. このプロジェクトの「全体要件」サマリ（MVP想定）

#### 1-1. ゴール

* 東京都内の「カーニボア向け肉料理が食べられる店」を、
  できるだけ機械的かつ高品質に抽出し、
  Google マイマップにインポートできる形（CSV/スプレッドシート）で出力する。

---

#### 1-2. スコープ

* 対象エリア: 東京都（23区＋多摩を含む）
* 対象カテゴリ: 肉メインの飲食店

  * 焼肉 / ホルモン / ジンギスカン / シュラスコ / ステーキ / 肉バル / ジビエ / ローストビーフ / しゃぶしゃぶ / すき焼き など
* 対象データソース（例）

  * ホットペッパーグルメAPI
  * （利用可能なら）その他グルメAPIやチェーン店の店舗一覧
  * ※ Google Places API はコスト面の理由で現在は利用停止。設計上は再接続できるように abstractions を維持。

---

#### 1-3. 機能要件（パイプライン）

1. **東京都のグリッド生成**

   * 東京都を緯度経度のバウンディングボックスで定義。
   * 半径 500〜1000m の円でグリッド分割し、各中心点のリストを作る。

2. **飲食店候補の収集**

   * ホットペッパー API:

     * 東京エリアコード＋肉ジャンルの組合せで、全店舗をページング取得。
     * 追加で肉ジャンル別キーワード（焼肉専門店／牛タン／ジンギスカン／しゃぶしゃぶ／焼き鳥／焼きとん／サムギョプサル／シュラスコ／アメリカンBBQ 等）で網羅的に検索し、日本系〜多国籍の肉特化店を拾う。
   * 他API/チェーン一覧:

     * 利用する場合は同様に ingest 処理を用意。
   * （参考）Google Places API:

     * コスト都合で現在は呼び出しを停止。過去の Nearby/Text Search 方針は残し、再開時にグリッド検索を復活させる。

3. **データ格納**

   * 共通スキーマを持つ DB / ファイルに保存（例: PostgreSQL / SQLite / Parquet / 単純な CSV 群など）。
   * 主なフィールド（例）:

     * store_id（内部ID）
     * name, name_kana
     * address, prefecture, city
     * lat, lng
     * phone
     * genre_primary, genre_secondary
     * description / catch_copy
     * budget_lunch, budget_dinner
     * rating, review_count
     * sources（例: `["hotpepper"]` や 将来追加するソース名の集合）
     * external_ids（hotpepper_id や 他APIのID など）

4. **重複統合（マージ）**

   * 電話番号、緯度経度の近接、正規化した店名を使って「同一店」を判定。
   * 複数ソースの情報を一つのレコードに統合（rating はどちらか優先などのポリシーを決める）。

5. **カーニボアスコア算出**

   * 入力:

     * ジャンル / カテゴリ
     * 店名・キャッチコピー・説明文
     * メニュー情報（取れる範囲で）
     * rating / review_count
   * 出力:

     * carnivore_score（数値）
     * carnivore_rank（S/A/B など）
   * 初期バージョンはルールベース:

     * 肉特化ジャンル（焼肉, シュラスコ等）なら大きく加点。
     * 「食べ放題」「肉盛り」「塊肉」などのキーワードで加点。
     * rating・件数で品質補正。
   * 一定以上のスコアのみマップに載せる（例: S/A のみ）。

6. **品質フィルタリング**

   * 営業ステータス（閉店フラグなど）で除外。
   * rating が極端に低い店舗の除外。
   * 住所・座標が東京都外のものを除外。

7. **Google マイマップ用の出力**

   * CSV or スプレッドシート形式で出力。
   * カラム例:

     * name, address, lat, lng
     * carnivore_rank
     * genre
     * rating, review_count
     * url（公式 or グルメサイト）
     * notes（カーニボア向きコメント）
   * マイマップ側で「色＝ランク」「アイコン＝ジャンル」でスタイリングできるようにする。

8. **更新・同期（任意 / できれば）**

   * バッチスクリプトとして:

     * 定期的に各 API から情報を更新。
     * カーニボアスコア再計算。
     * 閉店・新店を反映した CSV を再生成。

概要:
ユーザーの現在地や指定エリア周辺の飲食店・食品スポットを地図上に表示するシンプルなウェブサイトです。例えば「肉マップ」として、近隣の精肉店や焼肉店をマッピングしたサイトを作成します。地図表示と位置情報取得の実装にGPTのコード生成を活用し、短期間でサイト構築を体験します。小規模ながら地域に特化したニッチサイトとして運営し、広告収入などで月数百～千円の収益化を目指します。

手順:

アイデアとデータ整理: マップで可視化するテーマを決めます（例：「徒歩圏内で買える高評価の肉料理店マップ」や「地元のおすすめ精肉店マップ」など）。次に表示するスポットのデータを準備しましょう。最初は5～10件程度、店舗名・住所・簡単な説明・写真URLなどをリストアップします。データは静的なJSONファイルにするか、Googleスプレッドシートにまとめて後で読み込む方法が手軽です（スプレッドシート公開でJSON取得可能）。

マップ表示のベース実装: ChatGPT(Codex)に依頼して、地図を表示するHTML/JavaScriptコードのひな形を作成します。具体的には「OpenStreetMapとLeaflet.jsを使って地図を表示し、指定座標にマーカーをプロットするコードを書いて」とプロンプトを出せばOKです。あるいはGoogle Maps APIを使う場合、「Google Maps埋め込み用のHTMLとJSコードを作成して、中心を〇〇に設定」と依頼します。Codexはこのようなふんわりした指示でも適切に解釈してコード化してくれるので、まずはAIにひな形を生成させます。

位置情報取得とマーカー描画: 続いて、ユーザーの現在地を取得して地図の中心に設定する機能を追加しましょう。Codexに「ブラウザのGeolocation APIを使って現在位置を取得し、地図をその位置にズームするコードを追記して」と頼むと、navigator.geolocationを用いたコードを提案してくれます。次に、手順1で用意したスポット一覧を地図上にマーカー表示する機能を実装します。スポットデータがJSONやスプレッドシートにある場合は、その読み込み（AjaxやfetchでJSON取得）もAIにコードを書かせます。「用意したスポット一覧JSONを取得し、各地点にマーカーを置くループ処理を実装して」といった具合です。各マーカーには店舗名や説明を吹き出し表示（ツールチップ）できるようGPTに指示しましょう。

コーディングとデバッグ: GPTから提案されたコードをエディタに貼り、実際にブラウザで動作確認します。初回で完璧に動かなくても心配ありません。不具合やエラーが出たら、そのメッセージや現象を再度ChatGPTに伝えて修正案をもらいます（「エラーXが出たので直して」などとプロンプト）。Codexはバグの原因を指摘し具体的な修正コードを提示してくれるので、対話しながら完成度を高めます。必要に応じて日本語でコメントを付けるよう依頼し、コード内容を理解する学習機会にしましょう。

デプロイ（公開）: ローカルでマップが正しく表示できたら、サイトをインターネット上に公開します。無料で簡単なのはGitHub Pagesです。Gitリポジトリを用意し、HTML/CSS/JS一式をコミットしたら、GitHubのリポジトリ設定でPagesを有効にします。これでhttps://ユーザ名.github.io/リポジトリ名/でサイトが公開されます。CodexはGitHub連携も得意で、Pull Request作成までサポートしてくれるので活用しましょう。更新があればGitにプッシュするだけでサイト反映されます。

マネタイズ設定: サイト公開後、少額でも収益化してみましょう。最も手軽なのはGoogle AdSenseです。審査に通れば広告コードをページに埋め込むだけで表示できます。特にGoogle Maps APIで地図を実装した場合、地図上にAdSense広告を表示することも可能です。Google Mapsを独自に埋め込めば（標準のiframeではなくAPIを使用する実装により）、地図上部に広告ユニットを表示してコンテンツやユーザー位置に合った広告を自動表示できます。Leafletなど他の地図でも、ページ内にバナー広告を配置すれば十分でしょう。また、紹介スポットに関連するアフィリエイトがあればリンクを貼ることも検討できます。例えば各店舗のオンライン通販があればその紹介で成果報酬を狙えますが、難しければ単に**「〇〇マップ提供：○○（あなたの名前）」**とサイト内に記し、「開発者を支援する」ボタン（Ko-fiやBuyMeACoffeeなど）を置いて寄付を募るのも一案です。

必要スキル: 基本的なHTML/CSSの知識と、ごく初歩的なJavaScriptの理解。Googleアカウント（マップAPIやスプレッドシート利用時）やGitHubのアカウント操作。コーディング自体はAIが補完するので専門知識は最低限でOKです。英語の簡単なAPIドキュメントを読む場面もありますが、AIに和訳や要約させれば対応可能です。

想定作業時間: 約2～4週間（平日夜や週末に少しずつ作業した場合）。初期設定と学習に1週間、マップ機能の実装・デバッグに1～2週間、公開と広告設定に数日というイメージです。AIを使うことで、ゼロから自力で実装するより格段にスピードアップできます。

マネタイズ方法: 主にページ上の広告収入です。Google AdSenseをサイトに設置し、ページビューに応じて収益を得ます。ニッチなローカル情報でも、月数千PV集まれば1,000円程度の広告収入になる可能性があります。例えば月間7,000PVで1,000円程度という報告もあります。最初はPV数が少ないと思いますが、SNSで地元の人にシェアしたりするとアクセスが増え、目標金額に近づくでしょう。そのほか、関連するアフィリエイト（あれば）や寄付受付によるわずかな収入も組み合わせて、月1,000円達成を狙います。
