### 1. このプロジェクトの「全体要件」サマリ（MVP想定）

#### 1-1. ゴール

* 東京都内の「カーニボア向け肉料理が食べられる店」を、
  できるだけ機械的かつ高品質に抽出し、
  Google マイマップにインポートできる形（CSV/スプレッドシート）で出力する。

---

#### 1-2. スコープ

* 対象エリア: 東京都（23区＋多摩を含む）
* 対象カテゴリ: 肉メインの飲食店

  * 焼肉 / ホルモン / ジンギスカン / シュラスコ / ステーキ / 肉バル / ジビエ / ローストビーフ / しゃぶしゃぶ / すき焼き など
* 対象データソース（例）

  * ホットペッパーグルメAPI
  * （利用可能なら）その他グルメAPIやチェーン店の店舗一覧
  * ※ Google Places API はコスト面の理由で現在は利用停止。設計上は再接続できるように abstractions を維持。

---

#### 1-3. 機能要件（パイプライン）

1. **東京都のグリッド生成**

   * 東京都を緯度経度のバウンディングボックスで定義。
   * 半径 500〜1000m の円でグリッド分割し、各中心点のリストを作る。

2. **飲食店候補の収集**

   * ホットペッパー API:

     * 東京エリアコード＋肉ジャンルの組合せで、全店舗をページング取得。
     * 追加で肉ジャンル別キーワード（焼肉専門店／牛タン／ジンギスカン／しゃぶしゃぶ／焼き鳥／焼きとん／サムギョプサル／シュラスコ／アメリカンBBQ 等）で網羅的に検索し、日本系〜多国籍の肉特化店を拾う。
   * 他API/チェーン一覧:

     * 利用する場合は同様に ingest 処理を用意。
   * （参考）Google Places API:

     * コスト都合で現在は呼び出しを停止。過去の Nearby/Text Search 方針は残し、再開時にグリッド検索を復活させる。

3. **データ格納**

   * 共通スキーマを持つ DB / ファイルに保存（例: PostgreSQL / SQLite / Parquet / 単純な CSV 群など）。
   * 主なフィールド（例）:

     * store_id（内部ID）
     * name, name_kana
     * address, prefecture, city
     * lat, lng
     * phone
     * genre_primary, genre_secondary
     * description / catch_copy
     * budget_lunch, budget_dinner
     * rating, review_count
     * sources（例: `["hotpepper"]` や 将来追加するソース名の集合）
     * external_ids（hotpepper_id や 他APIのID など）

4. **重複統合（マージ）**

   * 電話番号、緯度経度の近接、正規化した店名を使って「同一店」を判定。
   * 複数ソースの情報を一つのレコードに統合（rating はどちらか優先などのポリシーを決める）。

5. **カーニボアスコア算出**

   * 入力:

     * ジャンル / カテゴリ
     * 店名・キャッチコピー・説明文
     * メニュー情報（取れる範囲で）
     * rating / review_count
   * 出力:

     * carnivore_score（数値）
     * carnivore_rank（S/A/B など）
   * 初期バージョンはルールベース:

     * 肉特化ジャンル（焼肉, シュラスコ等）なら大きく加点。
     * 「食べ放題」「肉盛り」「塊肉」などのキーワードで加点。
     * rating・件数で品質補正。
   * 一定以上のスコアのみマップに載せる（例: S/A のみ）。

6. **品質フィルタリング**

   * 営業ステータス（閉店フラグなど）で除外。
   * rating が極端に低い店舗の除外。
   * 住所・座標が東京都外のものを除外。

7. **Google マイマップ用の出力**

   * CSV or スプレッドシート形式で出力。
   * カラム例:

     * name, address, lat, lng
     * carnivore_rank
     * genre
     * rating, review_count
     * url（公式 or グルメサイト）
     * notes（カーニボア向きコメント）
   * マイマップ側で「色＝ランク」「アイコン＝ジャンル」でスタイリングできるようにする。

8. **更新・同期（任意 / できれば）**

   * バッチスクリプトとして:

     * 定期的に各 API から情報を更新。
     * カーニボアスコア再計算。
     * 閉店・新店を反映した CSV を再生成。

---

#### 1-4. 非機能要件

* 自動化:

  * 1コマンド or 1スクリプト実行で「取得→統合→スコアリング→出力」まで流れる。
* 再現性:

  * APIキーなどは `.env` や設定ファイルから読み込む。
  * 依存パッケージは `requirements.txt` or `pyproject.toml` に明示。
* コスト・制限:

  * API のレート制限・課金を意識した設計（キャッシュやリトライ戦略）。
* デバッグ性:

  * 中間生成物（「生の API レスポンス」「マージ前後」「スコア計算結果」など）を確認しやすいようにログやサンプル出力を用意。

  * `AGENTS.md`（Codex 用の指示）

    * `export.py`（マイマップ出力）
    * `cli.py`（全体をつなぐエントリポイント）
  * `tests/`（各モジュールのテスト）
